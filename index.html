1. Overview

This guide provides B2B Partners with the essential information needed to integrate with Entain’s casino games platform. The integration allows partners to embed Entain’s casino games into their gaming portals and utilize backend features including bonus/reward systems, coin economy, lobby management, and jackpots.

1.1. Definitions

Player

A user of the casino service, registered with the B2B Partner.

Shadow User

Tracking identifier of the player, registered with the Casino service by the B2B Partner. Only enough details are saved by this service to ensure operation of games.

Channel Client

The type of client (web browser, mobile app) used by the player to access games. Games may be limited to one or more channels.

Game Session

Logical identifier for a player’s game instance.

Game Round

Individual betting round within a game session.

Wallet

Player’s account balance and transaction system.

Game

A single casino or other game being offered to the B2B Partner’s players via Entain’s Casino Service.

Lobby Feed

A catalogue of games offered to a player, organised logically. The categories and their criteria are out of scope of this document.

Jackpot Feed

A list of jackpots (and sub-jackpots) available to be won by the player, along with their trigger critera and the games with which they are associated.

1.3. Service Interaction Flow

  1. B2B Partner Backend creates shadow users in the Casino Service

  2. Partner’s client application retrieves lobby information from Casino Service

  3. Player selects a game from the Partner Portal

  4. B2B Partner Backend gets the game URL from the Casino Service, then launches the game in the client

  5. During gameplay, the game calls the Casino service for wallet operations, which are proxied to the Partner Wallet Service.

  6. Partner Wallet Service handles all financial transactions and player authentication.

2. Service Implementation Notes

2.1. Security Requirements

  • mTLS Implementation: Both services must implement mutual TLS for secure authentication

  • HMAC Authentication: Additional message integrity and idempotency protection

  • JSON API Support: RESTful API with JSON payloads

2.2. Security Considerations

  • Exchange of secret keys for HMAC authentication

  • Certificate management for mTLS

  • Session key lifecycle and rotation procedures

  • IP address validation and geolocation

3. Part 1: Casino Service Integration

3.1. Process Overview

The Casino Service is implemented by Entain and called by your B2B Partner Backend. This service provides game information, lobby management, jackpot data, and game launch capabilities. Your integration involves calling these services to offer games to your players.

3.2. Implementation Steps

3.2.1. Step 1: Player Management

Shadow User Creation

Create shadow users in the Entain system for tracking purposes:

  • Provide player ID from your system

  • Player nickname for display

  • Language preference (defaults to label default if omitted)

  • Account currency (ISO 4217 format)

  • Country code (optional, defaults to label default)

Bonus Application

Apply bonuses to players on deposit:

  • Support bonus codes

  • Handle deposit amounts and currency

  • Process bonus awards

3.2.2. Step 2: Content and Game Management

Lobby Integration

Retrieve lobby feeds containing:

  • Default lobby feed for all users

  • User-specific lobby feeds

  • Game categories and subcategories

  • Available games per category

  • Game metadata and icons

Jackpot Information

Access jackpot data including:

  • Available jackpot groups and pools

  • Current jackpot amounts and values

  • Eligible games for each jackpot

  • Trigger conditions and award types

  • Player participation tracking

3.2.3. Step 3: Game Launch

Game Launch Process

  1. Player selects game from lobby

  2. Your client calls Entain’s game launch endpoint

  3. Receive game launch URL

  4. Redirect player to game

Important Note: The game launch endpoint is called directly by the player client, bypassing mTLS and HMAC authentication.

4. Part 2: Partner Wallet Service Implementation

4.1. Process Overview

You must implement the Partner Wallet Service that will be called by the Casino Service during gameplay. This service handles player authentication, wallet operations, and game state tracking. It’s the core of your financial and player management integration.

4.2. Implementation Steps

4.2.1. Step 1: Player Authentication

Session Authentication

Implement player session authentication that:

  • Validates player auth tokens generated by your system

  • Verifies player ID matches the authenticated session

  • Returns player information and current balance

  • Handles authentication errors appropriately

Critical: The player_id parameter cannot be trusted before authentication as it passes through multiple layers.

4.2.2. Step 2: Wallet Operations

Balance Management

Your service must provide:

  • Real-time balance lookups

  • Support for game-specific balances (optional)

  • Currency handling in 1/100 units (e.g., 1050 for $10.50)

  • Timestamp validation for synchronization

Transaction Processing
Game Round Purchase (Betting)

  • Debit funds from player wallet

  • Validate sufficient funds before processing

  • Handle transaction idempotency

  • Return updated balance after transaction

Game Round Results (Payouts)

  • Credit winnings to player wallet

  • Handle jackpot payouts separately

  • Process jackpot contributions

  • Mark rounds as closed when appropriate

  • Support win/loss result types

Transaction Cancellation

  • Reverse previous transactions

  • Return funds to player wallet

  • Maintain transaction audit trail

4.2.3. Step 3: Session Management

Game Session Tracking

  • Track game session lifecycle

  • Handle session end notifications (unreliable - implement your own timeout logic)

  • Maintain session state for audit purposes

  • Process session-related events
