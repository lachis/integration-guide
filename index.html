1. Overview

This guide provides B2B Partners with the essential information needed to integrate with Entain’s casino games platform. The integration allows partners to embed Entain’s casino games into their gaming portals and utilize backend features including bonus/reward systems, coin economy, lobby management, and jackpots.

1.1. Definitions

Player

A user of the casino service, registered with the B2B Partner.

Shadow User

Tracking identifier of the player, registered with the Casino service by the B2B Partner. Only enough details are saved by this service to ensure operation of games.

Channel Client

The type of client (web browser, mobile app) used by the player to access games. Games may be limited to one or more channels.

Wallet

Player’s account balance and transaction system.

Transaction

A financial operation (bet, win, bonus payout) that affects the player’s wallet balance.

Player Session

An authenticated session between the player and the wallet service, established during authentication and used for subsequent wallet operations.

Game Provider

The company that develops and supplies games to the Entain platform.

Game

A single casino or other game being offered to the B2B Partner’s players via Entain’s Casino Service.

Live Casino

Real-time casino games with live dealers, streamed to players.

Bingo Room

A virtual space where bingo games are conducted, with specific rules and player limits.

Game Round

Individual betting round within a game session.

Game Session

A period of continuous gameplay by a player, containing one or more game rounds.

Bonus

A promotional reward given to players, which can be cash, free spins, free tickets, or other non-monetary benefits.

Prebuy

The ability for bingo players to purchase tickets in advance for future game rounds.

Sticker

Visual promotional labels displayed on games in the lobby (e.g., "New", "Exclusive", "Hot").

Lobby Feed

A catalogue of games offered to a player, organised logically. The categories and their criteria are out of scope of this document.

Jackpot Pool

A shared prize fund that accumulates across multiple games or players until won.

Jackpot Feed

A list of jackpots (and sub-jackpots) available to be won by the player, along with their trigger critera and the games with which they are associated.

MTLS

Mutual Transport Layer Security - bidirectional authentication where both client and server verify each other’s certificates.

HMAC

Hash-based Message Authentication Code - cryptographic method for verifying message integrity and authenticity.

1.3. Service Interaction Flow

  1. B2B Partner Backend creates shadow users in the Casino Service

  2. Partner’s client application retrieves lobby information from Casino Service

  3. Player selects a game from the Partner Portal

  4. Player client calls Casino Service directly to get game launch URL

  5. During gameplay, the game calls the Casino service for wallet operations, which are proxied to the Partner Wallet Service.

  6. Partner Wallet Service handles all financial transactions and player authentication.

2. Service Implementation Notes

2.1. Security Requirements

  • mTLS Implementation: Casino Service requires mTLS for backend-to-backend calls

  • HMAC Authentication: Additional message integrity protection for Casino Service

  • Player Client Calls: Game launch endpoint uses standard HTTPS only

  • JSON API Support: RESTful API with JSON payloads

2.2. Security Considerations

  • Exchange of secret keys for HMAC authentication

  • Certificate management for mTLS

  • Session key lifecycle and rotation procedures

  • IP address validation and geolocation

3. Part 1: Casino Service Integration

3.1. Process Overview

The Casino Service is implemented by Entain and called by your B2B Partner Backend. This service provides game information, lobby management, jackpot data, and game launch capabilities. Your integration involves calling these services to offer games to your players.

3.2. Implementation Steps

3.2.1. Phase 1: Player Setup

Initial Player Registration

When a player registers on your platform:

  1. Capture Player Information: Collect nickname, language, currency, and country during registration

  2. Create Shadow User: Call CreateShadowUser with player details to establish tracking in Entain system

  3. Store Entain Account ID: Save the returned entain_account_id for future reference

  4. Generate Auth Tokens: Create player authentication tokens for game launch

Bonus Setup

For promotional campaigns:

  1. Apply Welcome Bonuses: Use ApplyBonus with bonus codes during player onboarding

  2. Monitor Bonus Balances: Periodically call GetPlayerBonuses to display available bonuses

3.2.2. Phase 2: Game Discovery

Content Synchronization

  1. Game Catalog Updates: Call GetAvailableGames every ~30 minutes to refresh your game catalog

  2. Lobby Configuration: Use GetCommonLobby to retrieve game categories and organization

  3. Jackpot Information: Call GetJackpots every 30 seconds to display current jackpot values

  4. Live Updates: Call GetGameUpdates every 3-5 seconds for real-time lobby information

Frontend Integration

  1. Cache Game Data: Store game metadata locally to reduce API calls

  2. Display Lobby: Render games using categories and stickers from lobby feed

  3. Show Jackpots: Display jackpot values with automatic refresh

  4. Filter Content: Use game_types parameter to show casino, live casino, or bingo games

  5. Game Graphics: Fetch game tiles and visual assets from Entain’s image CDN. CDN base URL and image path construction details are defined separately

3.2.3. Phase 3: Game Launch

Player-Initiated Game Launch

When player selects a game:

  1. Validate Player Session: Ensure player is authenticated on your platform

  2. Prepare Launch Parameters: Gather required data (player_id, nickname, currency, user_ip, etc.)

  3. Direct Client Call: Player’s browser/app calls GetGameLaunchUrl directly (bypasses your backend)

  4. Handle Response: Redirect player to the returned game launch URL

  5. Session Tracking: Monitor game session for analytics (optional)

Critical: Game launch bypasses backend authentication - ensure your client properly validates player session.

3.2.4. Phase 4: Bingo Features

Prebuy Implementation

For bingo game integration:

  1. Display Available Slots: Call GetBingoPrebuySlots to show upcoming bingo games

  2. Handle Purchases: Use PrebuyBingoTickets when players buy advance tickets

  3. Process Payments: Route prebuy transactions through your wallet service

  4. Manage Inventory: Track purchased tickets and game participation

4. Part 2: Partner Wallet Service Implementation

4.1. Process Overview

You must implement the Partner Wallet Service called by Casino Service during gameplay. This handles player authentication, wallet operations, and transaction tracking.

4.2. Implementation Steps

4.2.1. Phase 1: Authentication Service

Player Session Validation

Implement PlayerSessionAuthenticate to handle game authentication:

  1. Token Validation: Verify the player_auth_token against your authentication system

  2. Player Lookup: Retrieve player information using validated token

  3. Session Creation: Generate and return a player_session_id for wallet operations

  4. Security Check: Validate that player_id matches the authenticated token

Critical: Never trust the player_id parameter before authentication - it passes through multiple untrusted layers.

4.2.2. Phase 2: Wallet Service Implementation

Core Wallet Operations

Implement these essential operations:

  1. PlayerGetBalance: Return real-time player balance for game display

  2. GameRoundTransaction: Process bets, wins, and jackpot payouts atomically

  3. GameRoundBetCancelled: Handle bet reversals and refunds

  4. BonusPayout: Process bonus awards from promotional campaigns

Transaction Processing Flow

For each game round:

  1. Receive Transaction Request: Game sends bet/win data via GameRoundTransaction

  2. Validate Session: Ensure player_session_id is valid and active

  3. Process Atomically: Handle all transactions in the request as a single unit

  4. Update Balance: Debit bets, credit wins, handle jackpot contributions

  5. Return Updated Balance: Send current balance back to game

  6. Audit Trail: Log all transactions for compliance and reconciliation

4.3. Entain B2B Wallet Service API

Provides services to authenticate a player, start game sessions and perform operations on the player’s wallet in response to game events. It must be implemented by the B2B Partner. The response expects the player balance, which could be game-specific if the game_id is provided.

Jackpot data stale, no jackpot updates

Specific provider games unavailable

6.2.5. Implementation Recommendations

Monitoring Strategy

  • Health Check Frequency: Poll health endpoints every 30-60 seconds

  • Alerting Thresholds: Alert on WARNING status, escalate on DEGRADED/DOWN

  • Dependency Timeouts: Set appropriate timeouts for each dependency check

  • Circuit Breakers: Implement circuit breakers for external dependencies
